MySQL 时间 '0000-00-00'异常的解决

教程整理
  • MySQL

在Java动态数据交互项目中，如数据录入不规范，查询MySQL数据库经常会遇到一个问题：
java.sql.SQLException:Value ‘0000-00-00’ can not be represented as java.sql.Date
这是JDBC将date类型的字段转成String类型时，遇到date类型字段为’0000-00-00’，转换不了，所以抛出该异常。

MySQL数据库的date、timestamp等时间类型定义时如果不指明为NULL，则默认为not null，产生一个默认值即’0000-00-00’。因此在输入对应数据时，应注意避免这种默认值的录入。如在准备数据或测试阶段出现上述情况该如何暂时避开这种异常呢？
有三种解决方案：

解决方案1

在项目配置文件中(通常为.properties文件)，数据库连接url增加参数zeroDateTimeBehavior=convertToNull或者round，可以将默认的’0000-00-00’转换为其他值；
想起之前还碰到过一个问题：以前在DB2中事件类型默认值是’1900-00-00’，在MySQL中却一直报错，经查询发现是MySQL的时间类型默认值必须从’1970-00-00’开始。

解决方案2

定义时间类型时给出默认值，例如 date not null default ‘1970-00-00’。

解决方案3

定义时间类型时默认为null，例如 date null。

备注
官网给出timestamp类型默认值default 范围是 ‘1970-01-01 00:00:01’ UTC to ‘2038-01-19 03:14:07’ UTC。
在中国，由于时区问题，timestamp默认值default范围则是’1970-01-01 08:00:01’ to ‘2038-01-19 11:14:07’ 。
